LU-19585 lraft: support Raft protocol

A new kernel module 'lraft' is added to support Raft protocol in
Lustre system, it contains four main parts:
1. generic Raft protocol code from https://github.com/willemt/raft
2. callbacks and RPC handlers for the above protocol
3. a kernel thread to finish Raft state machine, and Raft
   initialization and finish code
4. a new RPC to handshake with other MDTs/OSTs to get their Raft mount
   option and mgs_target_info

Usage:
* mount option 'raft' and 'raft_nonvote' is added to mount MDT as Raft
  voting node or as non-voting node
* the default value of 'raft_num_voting_nodes' (the number of Raft
  voting nodes) is '5', which can tolerate 2 nodes failure, and it can
  be set to '3' by below command if total MDT count is below 5, or
  user wants less voting nodes, in which case only one node failure
  can be tolerated:
        lctl set_param raft_num_voting_nodes=3
  the other values are not supported, this should be set on all MDTs
  to avoid confusion
* if MDT count is equal or more than 5, it's suggested to mount 5 MDTs
  as Raft voting nodes, and 3 MDTs (if there are) as non-voting nodes,
  and the rest not mounted as Raft nodes. If the MDT count is 3 or 4,
  it's suggested to have 3 voting nodes, and the rest as non-voting
  node. If the MDT count is below 3, MDTs shouldn't be mounted as Raft
  node. The actual voting nodes should match the
  'raft_num_voting_nodes' set, otherwise the Lustre system may become
  inaccessible (if services like FLDB are built upon Raft in the
  future).
* user can get raft state by 'lctl get_param -n mdt..raft'

Ideally if a system is built upon Raft, the Raft cluster should be
started first, and elect the leader, then the leader can provide
service to the system.
However Lustre server startup is tightly bond with MGS, so it's hard
to start Raft cluster before enabling MGS service, and MGS service has
similar membership management as Raft protocol, as a result, Lustre
doesn't use Raft protocol to manage membership, but use it for leader
election and FSM (finite state machine).

Lustre system now starts as below:
1. specify a starter MGS (which could be any MDT, though normally
   MDT0, or a separate MGS) to start MDTs and OSTs in the old way
2. if an MDT is mounted as Raft node, it handshakes with other
   MDTs/OSTs to get their mgs_target_info, and add Raft nodes if the
   peer MDTs are mounted as Raft nodes (OST can't be Raft node)
3. if there are more than (raft_num_voting_nodes / 2) voting nodes
   connected, start leader election

Updated test-framework.sh to mount MDT0 to MDT4 as Raft voting nodes,
and MDT5 to MDT7 as non-voting ndoes, and if total MDT count is 3 or
4, mount MDT0 to MDT2 as voting nodes, and the rest as non-voting
node. MDTs won't be mounted as Raft nodes if total MDT count is 1 or
2.
